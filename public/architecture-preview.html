Architecture Preview

Welke route is nu het slimst voor offline-first en de gedeelde database?

De plan-site stuurt aan op één gedeelde Core API met PostgreSQL voor de Service App, Vanventory en OCR. Daarom is een tijdelijke mock-oplossing minder interessant dan een kleine maar echte fundering die later herbruikbaar blijft.

Optie A

Tijdelijk verder op mock data

Snel voor een demo-fix, maar de offline-flow moet later opnieuw omgebouwd worden zodra de echte backend er is.

Pluspunten

  • Snelste korte route voor alleen zichtbare UI-fixes.
  • Geen database of backend nodig om te starten.

Minpunten

  • Offline-sync moet later opnieuw gebouwd worden.
  • Botst met het masterplan van een gedeelde databron.
Optie B

Eigen database alleen voor deze app

Beter dan mock data, maar deze app krijgt dan zijn eigen backendpad terwijl het plan net een gedeelde Core API voorziet.

Pluspunten

  • Sneller dan een volledige Core API opzetten.
  • Je hebt meteen echte opslag voor deze app.

Minpunten

  • Architectuur wijkt af van het ERP-masterplan.
  • Andere apps kunnen deze laag later niet netjes hergebruiken.
Optie C

Minimale gedeelde Core API + PostgreSQL

Aanbevolen

De service-app blijft offline-first, maar leest straks uit dezelfde gedeelde databasis als de andere apps. De huidige mock data gebruiken we als seed.

Pluspunten

  • Past bij plan.bossuyt.fixassistant.com.
  • Offline-sync maar één keer juist bouwen.
  • Vanventory en OCR kunnen later dezelfde bron gebruiken.

Minpunten

  • Eerst wat funderingswerk voordat de app volledig overschakelt.
Aanbevolen pad

Eerst de gedeelde fundering, daarna de app erop aansluiten

Dit houdt de architectuur proper en voorkomt dat we offline-sync, routing en data-opslag straks opnieuw moeten uitvinden.

1

Reistijd-bug nu herstellen zodat start/einde meteen de route herberekenen.

2

Minimale PostgreSQL + Drizzle schema opzetten voor techniekers, klanten, sites en werkorders.

3

Huidige mock data als seed laden zodat staging meteen echte records heeft.

4

Sync-endpoint bouwen dat deze service-app leest en naar IndexedDB cachet.

5

Offline-cache op 10 volledige werkbonnen brengen en later Vanventory/OCR laten aansluiten.

Datastroom

Service App → Core API → Shared PostgreSQL

Service App

Day view, werkbonnen, IndexedDB en offline-first ervaring voor de technieker.

Core API

Gedeelde backendlaag met sync, routing, auth en businessregels.

Shared PostgreSQL

Eén bron van waarheid voor klanten, sites, techniekers en werkorders.

Later ook

Vanventory en OCR kunnen zonder extra datamigratie op dezelfde bron aansluiten.
